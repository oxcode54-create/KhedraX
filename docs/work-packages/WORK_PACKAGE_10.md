## 1. Engineering Objective

Add the registry entries needed to generate all eight portfolio agents
below, using **only** existing registry mechanisms
(`agentTypes/`, `personas/`, `modules/`) and **only** the existing
`in-memory` memory backend. Generate each portfolio agent as a real,
standalone project via `createAgent()`, and turn each generation into a
permanent integration test.

**The strongest possible proof of this package's stated objective —
"architecture scales through registries, not engine changes" — is zero
changed files under `src/`.** Every deliverable in this package should be
achievable by adding new `.json` and `.md` files under `agentTypes/`,
`personas/`, and `modules/`, plus new test files. If you find yourself
needing to change even one `.ts` file to make a portfolio agent generate
correctly, stop and explain exactly why before making the change — that's
the one condition under which this package's scope expands, per the
Constitution and this package's own explicit instruction.

## 2. Scoping Decision Already Made — Read Before Starting

**Memory backend:** every portfolio agent uses the default `in-memory`
backend. No new memory backend is needed, and no `--memory-backend` CLI
flag is being added in this package — Memory Engine only ever scaffolds
configuration/documentation regardless of backend choice (it never
implements runtime storage, per its ownership boundary), so backend choice
has no bearing on whether any portfolio agent generates correctly.

**Workflow templates and deployment targets:** `AgentDNA.workflows` and
`AgentDNA.deployment` remain empty `{}` for every portfolio agent, exactly
as they have been for every agent generated by every prior work package.
Building real `workflowTemplates/`/`deploymentTargets/` registries that a
Template Engine (or a new engine) would need to consume is a genuine new
capability, not a registry-data addition to an existing consumer — it is
explicitly **out of scope** for this package and would require its own,
separately-scoped work package. Do not attempt to populate these DNA
sections in this package.

**Two of the eight portfolio agents need a genuinely new module** (DevOps
Automation Agent, Social Media Manager) — neither maps cleanly onto any
existing module (`memory`, `discord`, `email`, `github`, `rag`). Adding a
new module is exactly the kind of registry growth Work Package #4 already
established as safe and expected — it is not the "engine change" this
package is guarding against.

## 4. Worked Example 1: an agentType + persona pair (Discord Moderator)

Follow this exact shape for all six new agentTypes and seven new personas
listed in §5 — this is not a special case, just the first fully-specified
one.

**`agentTypes/discord-moderator/agentType.json`:**
```json
{
  "name": "discord-moderator",
  "version": "1.0.0",
  "defaultModules": ["discord", "memory"],
  "persona": { "presetName": "vigilant-moderator" }
}
```

**`personas/vigilant-moderator/persona.json`:**
```json
{
  "name": "vigilant-moderator",
  "version": "1.0.0",
  "tone": "firm",
  "traits": ["consistent", "fair", "de-escalating"],
  "constraints": [
    "Never ban a user without following the server's documented escalation policy.",
    "Never publicly shame a user; handle moderation actions discreetly."
  ],
  "escalationPolicy": "Escalate to a human moderator after a user's third rule violation within 30 days."
}
```

**Research's additive persona default** — `research`'s `agentType.json`
currently has no `persona` field at all (unlike `customer-support`, which
already has one). Adding one now is a pure addition, not a change to
existing behavior, and is safe under `VERSIONING_POLICY.md` §2:
```json
{
  "name": "research",
  "version": "1.0.0",
  "defaultModules": ["rag"],
  "persona": { "presetName": "rigorous-researcher" }
}
```

## 6. Two New Modules (exact same file shape as Work Package #4's discord/email/github/rag)

**`modules/infrastructure/module.json`:**
```json
{
  "name": "infrastructure",
  "version": "1.0.0",
  "capabilities": [
    "Trigger and monitor CI/CD pipeline runs.",
    "Query infrastructure state (deployments, services, health checks)."
  ],
  "constraints": [
    "Never apply a destructive infrastructure change without explicit confirmation.",
    "Never modify production credentials or secrets directly."
  ]
}
```
`configuration/default.json`: `{ "environment": "staging", "requireConfirmationForProd": true }`
`prompts/fragment.md`: `This module provides infrastructure automation scaffolding: triggering CI/CD pipelines and querying deployment/service state.`
`prompts/fragment.meta.json`: `{ "section": "instructions", "priority": 0, "exclusive": false }` (matches every existing module's default)
`implementation/README.md` / `tests/README.md`: same "v1 scaffold — configuration and prompt fragment only; no runtime implementation yet" wording pattern established for the four Work Package #4 modules (not "placeholder," per Constitution #10).

**`modules/social-media/module.json`:**
```json
{
  "name": "social-media",
  "version": "1.0.0",
  "capabilities": [
    "Draft and schedule posts across configured social media channels.",
    "Monitor mentions and comments for engagement opportunities."
  ],
  "constraints": [
    "Never publish a post without it passing the configured brand style checklist.",
    "Never delete a public comment except per the documented moderation policy."
  ]
}
```
`configuration/default.json`: `{ "channels": [], "requireApprovalBeforePosting": true }`
`prompts/fragment.md`: `This module provides social media scaffolding: drafting/scheduling posts and monitoring engagement across configured channels.`
Same `fragment.meta.json` and README wording pattern as `infrastructure` above.

## 7. Required Tests

One integration test per portfolio agent (8 total, in a new
`tests/unit/portfolio.test.ts`, reusing the fixture-copy pattern already
established), each: calls `createAgent()` for real with that agent's exact
`--type`/`--modules` combination, and asserts:
1. `agent.yaml` contains the expected `agent.type`, `persona.presetName`
   (where applicable), and `modules` list.
2. `docs/README.md` contains that agent's persona tone and every selected
   module's capability and constraint entries (reuse Work Package #5's
   assertion style).
3. `prompts/README.md` contains every selected module's fragment content,
   correctly composed (reuse Work Package #3's assertion style).
4. Generation succeeds with zero errors and the standalone guarantee holds
   (no `khedrax` string anywhere in the output).
